Orders

Order

What the client asked for

An order is a client's request for a delivery, collection, or service at a specific site. It captures what needs to happen — not how it will be done. An order has no knowledge of vehicles, routes, or scheduling. That comes later.

Key points

  • Every order specifies a site (where), a time window (when), and what needs to be delivered, collected or done.
  • For deliveries and collections, an auxiliary site is also specified — typically the depot where goods are loaded or offloaded.
  • An order can be too large for a single vehicle. In that case, it is split into multiple jobs when scheduling.
  • Orders are not sent to the scheduler directly — jobs are.
States New → Planned → In Progress → Closed

Jobs

Job

Broken down for scheduling

A job takes an order (or part of an order) and turns it into something that can be scheduled. When a planner reviews orders, they create jobs that account for practical constraints — particularly vehicle capacity. Jobs are what the scheduler works with.

Key points

  • One order can produce multiple jobs — for example, if the order is too large for a single vehicle, it is split into multiple smaller jobs.
  • A job can also combine parts of multiple orders if they share the same site and time window.
  • A job does not have to be linked to an order — it can exist independently if needed.
  • Jobs are what get grouped into trips by the scheduler.
States New → Planned → In Progress → Closed

Trips

Trip

The vehicle's route

A trip is a sequence of planned site visits (stops) assigned to a vehicle. It is created when the scheduler groups jobs together and builds a route. A trip has a start site, a start time, and a list of stops in order.

Key points

  • A trip can be allocated to a specific vehicle, a virtual vehicle (when the exact vehicle is not yet known), or left unallocated if imported from an external system.
  • The start site is typically the vehicle's home depot. If the vehicle is already out from a previous trip, the start site will be the end of that trip.
  • The start time is the earliest the vehicle can depart and still reach all stops within their time windows.
  • A trip's requirements are the combined requirements of all its jobs — useful for validating that the right vehicle is allocated.

Stops & Activities

Stops and activities are the execution layer of a trip — where the work actually happens.

Stop

A planned site visit

A stop is a planned visit at a site as part of a trip. Each stop records the planned arrival time, any wait time (if the vehicle arrives before the site opens), travel distance and duration from the previous stop, and the activities to be carried out there.

States Planned → Arrived → In Progress → Completed
Activity

The work done at a stop

An activity is the fulfillment of a job during a stop. It is where the actual loading, offloading, or service takes place.

Key points

  • A delivery job produces two activities — one loading activity at the depot and one offloading activity at the delivery site.
  • A collection job works in reverse — one collection activity at the pickup site and one offloading activity at the depot.
  • A service job produces one activity at the service site.

Putting It All Together

A practical example showing how an order moves through the system from request to execution.

Scenario

Pick n Pay Rosebank orders 500 boxes of long-life milk to be delivered from JHB Depot on Tuesday morning between 08:00 and 12:00.

Order An order is created: deliver 500 boxes of long-life milk to PnP Rosebank from JHB Depot, time window Tuesday 08:00–12:00. The order captures the client's requirement — nothing else.
Job The planner reviews the order. 500 boxes fit in one 4-ton vehicle, so one job is created from the order. If the order had been for 1,000 boxes requiring two vehicles, two jobs would have been created.
Trip The scheduler creates a trip for a 4-ton vehicle: start at JHB Depot at 07:30, visit PnP Rosebank, return to depot. The trip is allocated to vehicle BC12DF GP.
Stops The trip has two stops: JHB Depot (start, loading) and PnP Rosebank (delivery, offloading).
Activities Two activities are created: a loading activity at JHB Depot (500 boxes onto the vehicle) and an offloading activity at PnP Rosebank (500 boxes off the vehicle).

Terminology

Tramm uses specific terms consistently throughout the system. These definitions apply everywhere.

Term Meaning
Entity A data record stored in the system with a unique identifier. Examples: Order, Job, Trip, Vehicle, Site.
Entity Class A template for an entity. If a class field is updated, all instances inherit the new value unless they have overridden it. Example: Vehicle Class defines defaults that individual vehicles can inherit or override.
Complex Type A structured field type with sub-fields, similar to an entity, but without its own unique identifier. It exists as part of another entity. Example: a Stop is a complex type within a Trip.
Field * An asterisk next to a field name means it is required — it must be given a value when the record is created.
Field ^ A caret means the field is overridable. If left blank, it inherits its value from the entity class.
Field + A plus means the field is augmentative. Its value is combined with the value from the class (by concatenation or union) rather than replacing it.
Time Window A specific period with a From and To date/time. Used to specify when an order must be serviced, when a site is available, etc.
Operating Hours A recurring weekly availability schedule. For example, a site that is open Monday–Friday 07:00–16:00. Used for vehicles, sites, and operators.
Partition An isolated data container within the system. Data in one partition is not visible in another. Used to separate different customers or departments operating on the same system.
